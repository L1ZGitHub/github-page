Gemini Embedding 2 changes the calculus for multimodal RAG in a fundamental way: instead of maintaining separate retrieval pipelines for each modality and fusing results at query time, you get a single shared embedding space where text, images, video segments, and audio all live together. That means a text query can retrieve a diagram, a spoken explanation, or a video clip based on semantic proximity, not just metadata matches.

This article walks through how to build that kind of system practically, from ingestion to retrieval, with real code.

## What Gemini Embedding 2 Actually Offers

Google's multimodal embedding models (accessible via Vertex AI as `multimodalembedding@001`, with newer generations following the `gemini-embedding-*` naming) produce 1408-dimensional vectors by default, with optional 128, 256, and 512 dimension variants for trade-off tuning. The critical property is **cross-modal alignment**: a photo of a whiteboard containing an equation and a text description of that equation will have high cosine similarity in this space.

This contrasts sharply with dual-encoder approaches like CLIP, which align text and images but don't natively handle audio or video. Gemini's embedding stack extends alignment across four modalities, with video handled as sequences of frames with optional audio tracks.

Concretely, the API accepts:
- Raw text strings
- Image bytes or Google Cloud Storage URIs (JPEG, PNG, GIF, BMP, WEBP)
- Video URIs (MP4, MOV, AVI) with optional start/end offsets
- Audio files (WAV, MP3, FLAC, OGG) via recent model versions

All of these produce vectors in the same space. That's the foundation everything else builds on.

## Architecture Overview

A production multimodal RAG system with Gemini Embedding 2 has three distinct layers:

**Ingestion layer**: modality-specific preprocessing feeding a unified embedding pipeline. Text gets chunked; images get resized and encoded; videos get segmented into clips; audio gets transcribed or embedded directly. Each chunk gets embedded and stored with its source metadata.

**Retrieval layer**: a single vector index (or a set of namespace-partitioned indices) that accepts any modality as a query. The query itself gets embedded using the same model, then ANN search returns the top-k results regardless of their original modality.

**Generation layer**: a multimodal LLM (Gemini 1.5 Pro or Gemini 2.0) that receives the retrieved chunks alongside the original query and generates a response. For image and video chunks, the actual content is passed to the generator, not just metadata.

The key insight is that the retrieval layer doesn't care about modality. Once everything is a vector, the distance metric is universal.

### Embedding Each Modality

```python
from pathlib import Path
import base64

def embed_text(text: str, dimension: int = 1408) -> list[float]:
    embeddings = mm_model.get_embeddings(
        contextual_text=text,
        dimension=dimension,
    )
    return embeddings.text_embedding

def embed_image(image_path: str, context: str = "", dimension: int = 1408) -> list[float]:
    image = Image.load_from_file(image_path)
    embeddings = mm_model.get_embeddings(
        image=image,
        contextual_text=context,  # optional caption or alt-text
        dimension=dimension,
    )
    return embeddings.image_embedding

def embed_video_segment(
    video_uri: str,
    start_offset_sec: float = 0.0,
    end_offset_sec: float | None = None,
    dimension: int = 1408,
) -> list[float]:
    video = Video.load_from_file(video_uri)
    kwargs = {"start_offset_sec": start_offset_sec}
    if end_offset_sec is not None:
        kwargs["end_offset_sec"] = end_offset_sec
    
    embeddings = mm_model.get_embeddings(
        video=video,
        video_segment_config=vertexai.vision_models.VideoSegmentConfig(**kwargs),
        dimension=dimension,
    )
    return embeddings.video_embeddings[0].embedding
```

Notice I'm threading `contextual_text` into image embeddings. This is worth doing when you have captions, alt text, or surrounding paragraph context. The model fuses the text signal with the visual signal during embedding, which meaningfully improves retrieval precision for domain-specific images like technical diagrams.

## Building the Unified Index

I'll use ChromaDB here for simplicity, but the same logic applies to Pinecone, Weaviate, or pgvector.

```python
import chromadb
from chromadb.config import Settings

client = chromadb.PersistentClient(path="./multimodal_index")

collection = client.get_or_create_collection(
    name="multimodal_rag",
    metadata={"hnsw:space": "cosine"},
)

def ingest_document(
    content_id: str,
    embedding: list[float],
    modality: str,          # "text" | "image" | "video" | "audio"
    source_path: str,
    metadata: dict,
):
    collection.add(
        ids=[content_id],
        embeddings=[embedding],
        metadatas=[{
            "modality": modality,
            "source": source_path,
            **metadata,
        }],
    )
```

### Ingestion Pipeline for a Mixed Corpus

```python
import uuid
from pathlib import Path

def ingest_corpus(corpus_dir: str):
    corpus_path = Path(corpus_dir)
    
    # Text documents
    for txt_file in corpus_path.glob("**/*.txt"):
        chunks = chunk_text(txt_file.read_text(), chunk_size=512, overlap=64)
        for i, chunk in enumerate(chunks):
            emb = embed_text(chunk)
            ingest_document(
                content_id=f"{txt_file.stem}_{i}_{uuid.uuid4().hex[:8]}",
                embedding=emb,
                modality="text",
                source_path=str(txt_file),
                metadata={"chunk_index": i, "text": chunk},
            )
    
    # Images
    for img_file in corpus_path.glob("**/*.{jpg,png,webp}"):
        caption = extract_caption(img_file)  # from EXIF, filename, or nearby text
        emb = embed_image(str(img_file), context=caption)
        ingest_document(
            content_id=f"{img_file.stem}_{uuid.uuid4().hex[:8]}",
            embedding=emb,
            modality="image",
            source_path=str(img_file),
            metadata={"caption": caption},
        )
    
    # Videos: segment into 30-second clips
    for vid_file in corpus_path.glob("**/*.mp4"):
        duration = get_video_duration(str(vid_file))  # ffprobe or similar
        segment_length = 30.0
        start = 0.0
        segment_idx = 0
        
        while start < duration:
            end = min(start + segment_length, duration)
            emb = embed_video_segment(str(vid_file), start, end)
            ingest_document(
                content_id=f"{vid_file.stem}_seg{segment_idx}_{uuid.uuid4().hex[:8]}",
                embedding=emb,
                modality="video",
                source_path=str(vid_file),
                metadata={
                    "start_sec": start,
                    "end_sec": end,
                    "segment_index": segment_idx,
                },
            )
            start = end
            segment_idx += 1

def chunk_text(text: str, chunk_size: int = 512, overlap: int = 64) -> list[str]:
    words = text.split()
    chunks = []
    i = 0
    while i < len(words):
        chunk = " ".join(words[i:i + chunk_size])
        chunks.append(chunk)
        i += chunk_size - overlap
    return chunks
```

## Cross-Modal Retrieval

The retrieval step is where the unified embedding space pays off. A user query is embedded as text, and the search returns whatever is semantically closest, regardless of modality.

```python
def retrieve(
    query: str,
    top_k: int = 5,
    modality_filter: list[str] | None = None,
) -> list[dict]:
    query_embedding = embed_text(query)
    
    where_clause = None
    if modality_filter:
        where_clause = {"modality": {"$in": modality_filter}}
    
    results = collection.query(
        query_embeddings=[query_embedding],
        n_results=top_k,
        where=where_clause,
        include=["metadatas", "distances"],
    )
    
    hits = []
    for metadata, distance in zip(
        results["metadatas"][0],
        results["distances"][0],
    ):
        hits.append({
            "score": 1 - distance,  # cosine similarity
            **metadata,
        })
    
    return sorted(hits, key=lambda x: x["score"], reverse=True)
```

You can also query with an image instead of text:

```python
def retrieve_by_image(
    image_path: str,
    top_k: int = 5,
) -> list[dict]:
    query_embedding = embed_image(image_path)
    
    results = collection.query(
        query_embeddings=[query_embedding],
        n_results=top_k,
        include=["metadatas", "distances"],
    )
    
    hits = []
    for metadata, distance in zip(
        results["metadatas"][0],
        results["distances"][0],
    ):
        hits.append({"score": 1 - distance, **metadata})
    
    return sorted(hits, key=lambda x: x["score"], reverse=True)
```

This is particularly useful for [multimodal AI agents](/blog/multimodal-ai-agents-when-ai-sees-hears-and-acts) where the agent might take a screenshot and want to find related documentation or prior examples from an internal knowledge base.

## Generation with Retrieved Context

Once you have retrieval results spanning multiple modalities, you need to assemble them into a prompt for the generator. Gemini 1.5 Pro and Gemini 2.0 accept inline images and video in their context windows, which makes this straightforward.

```python
import google.generativeai as genai
from pathlib import Path

genai.configure(api_key="YOUR_GEMINI_API_KEY")
generator = genai.GenerativeModel("gemini-2.0-flash")

def build_multimodal_prompt(query: str, hits: list[dict]) -> list:
    parts = [f"Answer the following question using the provided context.\n\nQuestion: {query}\n\nContext:\n"]
    
    for i, hit in enumerate(hits):
        modality = hit["modality"]
        parts.append(f"\n[Source {i+1} - {modality}]")
        
        if modality == "text":
            parts.append(hit.get("text", ""))
        
        elif modality == "image":
            parts.append(f"Image from {hit['source']}:")
            img_data = Path(hit["source"]).read_bytes()
            parts.append({"mime_type": "image/jpeg", "data": img_data})
            if hit.get("caption"):
                parts.append(f"Caption: {hit['caption']}")
        
        elif modality == "video":
            # For video, include a description or keyframe
            parts.append(
                f"Video segment ({hit['start_sec']:.1f}s - {hit['end_sec']:.1f}s) "
                f"from {hit['source']}"
            )
    
    parts.append(f"\nAnswer:")
    return parts

def answer(query: str, top_k: int = 5) -> str:
    hits = retrieve(query, top_k=top_k)
    prompt_parts = build_multimodal_prompt(query, hits)
    response = generator.generate_content(prompt_parts)
    return response.text
```

## Embedding Dimension Trade-offs

The choice of embedding dimension matters more in multimodal systems than in text-only ones. At 1408 dimensions you get the best retrieval quality, but storage and ANN search costs scale accordingly. For a corpus with tens of millions of chunks, 256-dimensional embeddings reduce index size by ~80% with a modest accuracy penalty.

I typically run a dimension sweep on a held-out eval set before committing to a production dimension. For most enterprise knowledge bases under a few million documents, 512 dimensions hits a good balance. If you're already building on the patterns covered in [scaling RAG systems to millions of documents](/blog/scaling-rag-systems-to-millions-of-documents), the same partitioning strategies apply: partition by modality or by domain, then query the relevant partitions based on the query type.

## Audio Handling

Audio support in the multimodal embedding model is available but worth treating carefully. For long-form audio (podcasts, meeting recordings), transcribe first with Gemini or Whisper, then embed the transcript as text. Direct audio embedding works better for short clips where the acoustic properties matter, such as matching sound effects or identifying speakers.

```python
def ingest_audio(audio_path: str, transcript: str | None = None):
    if transcript:
        # Prefer text embedding of transcript for long-form audio
        emb = embed_text(transcript)
        metadata = {"transcript": transcript[:1000]}
    else:
        # Direct audio embedding for short clips
        # Note: requires the audio embedding capability on your model version
        audio_bytes = Path(audio_path).read_bytes()
        emb = embed_text(f"Audio file: {Path(audio_path).name}")  # fallback
        metadata = {}
    
    ingest_document(
        content_id=f"audio_{Path(audio_path).stem}_{uuid.uuid4().hex[:8]}",
        embedding=emb,
        modality="audio",
        source_path=audio_path,
        metadata=metadata,
    )
```

## Practical Considerations

**Batch embedding**: the Vertex AI API supports batching up to 250 items per request for text. For images and video, batch size is lower (around 5-10 depending on size). Saturating throughput requires async batching with exponential backoff.

**Re-ranking**: cosine similarity in a shared embedding space is good but not perfect for cross-modal retrieval. A cross-encoder re-ranker trained on your domain improves precision significantly. The challenge is that most cross-encoders are text-only, so you need either a multimodal re-ranker or a step that converts retrieved images to text descriptions before re-ranking.

**Metadata filtering**: don't underestimate the value of pre-filtering by metadata before ANN search. If your corpus has timestamps, document types, or access control metadata, filtering at the vector store level before the ANN search dramatically improves both precision and latency. This is especially relevant if you're mixing internal confidential documents with public content.

**Versioning**: when you upgrade embedding models, old vectors are no longer comparable with new ones. Plan for periodic re-indexing or maintain model-versioned namespaces. This is the same discipline needed for [production-ready RAG systems](/blog/building-production-ready-rag-systems) generally, but it's more expensive when your corpus contains large video files.

The architecture described here integrates naturally with the broader pipeline patterns from [Multimodal RAG 2026](/blog/multimodal-rag-2026-vision-and-text-for-state-of-the-art-pipelines), where cross-modal fusion at the retrieval stage can be combined with vision-language model re-ranking for even higher precision.

## Key Takeaways

- Gemini Embedding 2 produces vectors for text, images, video, and audio in a single shared embedding space, enabling true cross-modal retrieval without late fusion hacks
- A text query can retrieve images, video clips, or audio segments based on semantic proximity, not just keyword or metadata matches
- Thread `contextual_text` into image embeddings when you have captions or surrounding context — it measurably improves precision for domain-specific visuals
- Segment videos into fixed-length clips (15-60 seconds) before embedding; store start/end timestamps as metadata for playback-linked citations
- For long-form audio, transcribing first and embedding the transcript as text typically outperforms direct audio embedding
- Embedding dimension choice (128 to 1408) is a meaningful trade-off: run a dimension sweep on a held-out eval set before committing to production
- Re-ranking with a cross-encoder improves cross-modal precision but requires a multimodal or text-conversion step since most re-rankers are text-only
- When upgrading embedding model versions, re-index your corpus or maintain model-versioned namespaces — old and new vectors are not comparable
