## Why Unified Embeddings Change Everything for Multimodal RAG

Most multimodal RAG systems are two separate retrieval systems duct-taped together. You have a text index, an image index, and some routing logic that decides which one to query. This works, but it creates a fundamental problem: a user asking "show me the architecture diagram that explains this concept" gets split into a text query and an image query that never actually talk to each other. Relevance scores from different embedding spaces are not comparable, and fusing them is guesswork.

Unified embedding models solve this by mapping text, images, and other modalities into a single shared vector space. A sentence and a diagram describing the same concept end up geometrically close. This makes cross-modal retrieval natural rather than hacked together.

This article walks through building a cross-modal retrieval pipeline from scratch using unified embeddings, covering indexing, query construction, and the practical tradeoffs that matter at production scale.

## What Makes an Embedding Space "Unified"

A unified embedding model is trained with contrastive objectives across modalities. The canonical example is CLIP: image-text pairs are pulled together in embedding space during training, and mismatched pairs are pushed apart. The result is a space where `cosine_similarity(embed("a cat sitting on a couch"), embed(photo_of_cat_on_couch))` is high, even though one is text and the other is pixels.

More recent models extend this further. Nomic Embed Multimodal, ImageBind, and the Gemini embedding family all offer joint spaces that go beyond image-text to include audio, video, and structured data. If your use case involves more than images and text, [Gemini Embedding 2's unified multimodal retrieval](/blog/multimodal-rag-with-gemini-embedding-2-unifying-text-images-video-and-audio-retrieval) is worth examining for its coverage of video and audio modalities alongside text and images.

The practical implication: you can embed everything into one index and issue a single query vector regardless of the modality you're searching for.

## Building the Indexing Pipeline

The indexing side is straightforward once you pick your embedding model. Here is a minimal pipeline using `open-clip-torch` for embeddings and `qdrant-client` for the vector store:

```python
import open_clip
import torch
from PIL import Image
from pathlib import Path
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct
import uuid

model, _, preprocess = open_clip.create_model_and_transforms(
    "ViT-B-32", pretrained="openai"
)
tokenizer = open_clip.get_tokenizer("ViT-B-32")
model.eval()

client = QdrantClient(":memory:")
client.create_collection(
    collection_name="multimodal_docs",
    vectors_config=VectorParams(size=512, distance=Distance.COSINE),
)

def embed_text(text: str) -> list[float]:
    tokens = tokenizer([text])
    with torch.no_grad():
        features = model.encode_text(tokens)
        features /= features.norm(dim=-1, keepdim=True)
    return features[0].tolist()

def embed_image(image_path: str) -> list[float]:
    image = preprocess(Image.open(image_path)).unsqueeze(0)
    with torch.no_grad():
        features = model.encode_image(image)
        features /= features.norm(dim=-1, keepdim=True)
    return features[0].tolist()

def index_document(content: str | Path, metadata: dict):
    if isinstance(content, Path):
        vector = embed_image(str(content))
        modality = "image"
    else:
        vector = embed_text(content)
        modality = "text"

    client.upsert(
        collection_name="multimodal_docs",
        points=[PointStruct(
            id=str(uuid.uuid4()),
            vector=vector,
            payload={**metadata, "modality": modality},
        )],
    )
```

The key detail here is that both text and image documents go into the same collection with the same vector size. No routing, no separate indexes. You can filter by `modality` at query time if needed, but you do not have to.

## Cross-Modal Retrieval at Query Time

With everything in one index, retrieval is a single nearest-neighbor search:

```python
from qdrant_client.models import Filter, FieldCondition, MatchValue

def retrieve(
    query: str,
    top_k: int = 5,
    modality_filter: str | None = None,
) -> list[dict]:
    query_vector = embed_text(query)
    
    search_filter = None
    if modality_filter:
        search_filter = Filter(
            must=[FieldCondition(
                key="modality",
                match=MatchValue(value=modality_filter),
            )]
        )
    
    results = client.search(
        collection_name="multimodal_docs",
        query_vector=query_vector,
        limit=top_k,
        query_filter=search_filter,
    )
    
    return [
        {
            "id": hit.id,
            "score": hit.score,
            "modality": hit.payload["modality"],
            "payload": hit.payload,
        }
        for hit in results
    ]
```

A text query naturally surfaces both relevant text passages and relevant images. The modality filter is optional - sometimes you want only images, sometimes you want the best match regardless of type. That flexibility is what you give up when you maintain separate indexes.

## Handling the Score Calibration Problem

Even with a unified space, raw cosine similarity scores are not perfectly calibrated across modalities. In practice, text-to-text retrieval tends to produce higher peak scores than text-to-image retrieval, because text pairs trained with stronger supervision. If you are mixing modalities in a single result list and ranking purely by score, images may get systematically underranked.

A lightweight fix is per-modality score normalization. During indexing, maintain running statistics on score distributions for each modality pair (query type vs. document type). At retrieval time, normalize scores before merging:

```python
import numpy as np

class ModalityScoreNormalizer:
    def __init__(self):
        # Accumulated (mean, std) for (query_modality, doc_modality) pairs
        self.stats: dict[tuple, tuple[float, float]] = {}
    
    def update(self, query_mod: str, doc_mod: str, scores: list[float]):
        key = (query_mod, doc_mod)
        arr = np.array(scores)
        self.stats[key] = (float(arr.mean()), float(arr.std() + 1e-8))
    
    def normalize(self, query_mod: str, doc_mod: str, score: float) -> float:
        key = (query_mod, doc_mod)
        if key not in self.stats:
            return score
        mean, std = self.stats[key]
        return (score - mean) / std
```

This approach works well for production systems where you can collect score distributions from your own data. It is worth doing before you tune any downstream reranking.

## Enriching Images with Metadata

One of the most effective techniques for cross-modal retrieval is captioning images before indexing. Rather than relying entirely on visual embeddings, you generate a text description of each image and store it alongside the image embedding. This gives you two retrieval paths for each image:

- Direct visual similarity via image embedding
- Semantic similarity via caption embedding

```python
from transformers import BlipProcessor, BlipForConditionalGeneration

caption_model = BlipForConditionalGeneration.from_pretrained(
    "Salesforce/blip-image-captioning-base"
)
caption_processor = BlipProcessor.from_pretrained(
    "Salesforce/blip-image-captioning-base"
)

def generate_caption(image_path: str) -> str:
    image = Image.open(image_path).convert("RGB")
    inputs = caption_processor(image, return_tensors="pt")
    with torch.no_grad():
        output = caption_model.generate(**inputs, max_new_tokens=50)
    return caption_processor.decode(output[0], skip_special_tokens=True)

def index_image_with_caption(image_path: str, metadata: dict):
    caption = generate_caption(image_path)
    
    # Index by visual embedding
    index_document(Path(image_path), {**metadata, "caption": caption})
    
    # Also index by caption embedding, linked back to the same image
    index_document(caption, {**metadata, "source_image": image_path, "modality": "image_caption"})
```

This dual-indexing approach significantly improves recall for conceptual queries that match a diagram's meaning but not its visual appearance. The tradeoff is roughly 2x storage for image documents.

## Structuring the RAG Generation Step

Once you have a mixed set of retrieved documents, you need to pass them to a generator that can handle both modalities. Vision-language models like Qwen2-VL, LLaVA, or GPT-4o are the practical choices here. The context construction step matters more than people realize:

```python
def build_multimodal_context(results: list[dict], query: str) -> dict:
    text_chunks = []
    image_paths = []
    
    for result in results:
        if result["modality"] == "text":
            text_chunks.append(result["payload"].get("content", ""))
        elif result["modality"] in ("image", "image_caption"):
            image_path = result["payload"].get("source_image") or result["payload"].get("path")
            if image_path:
                image_paths.append(image_path)
    
    # Deduplicate images (caption and visual embedding may both retrieve the same image)
    image_paths = list(dict.fromkeys(image_paths))
    
    context_text = "\n\n".join(text_chunks)
    
    return {
        "query": query,
        "text_context": context_text,
        "images": image_paths[:3],  # Cap to avoid context overflow
    }
```

Capping image count is important. Vision-language models degrade in quality with too many images in context, and the cost per image token is typically much higher than text. Three to five images is a reasonable upper bound for most queries.

## Production Considerations

A few things I have run into deploying unified multimodal indexes:

**Embedding latency asymmetry.** Image embedding is consistently slower than text embedding, sometimes by 3-5x for larger ViT models. If your indexing pipeline mixes modalities in a single queue, image processing can create backpressure. Separate queues with independent workers for each modality type prevents this.

**Index size growth.** Image embeddings are the same size as text embeddings (512 or 1536 floats depending on model), but the payloads are larger because you often store captions, bounding box metadata, and source file references. Use a payload store separate from the vector index if your Qdrant or Weaviate instance is under memory pressure.

**Embedding model versioning.** If you update your embedding model, your entire index becomes stale. Unlike pure text RAG where you can re-embed incrementally, images may not be trivially available for re-processing if they came from document uploads. Version your embedding model in collection metadata and build a re-indexing job before you need it.

For the generation side of the pipeline, [building production-ready RAG systems](/blog/building-production-ready-rag-systems) covers reranking and context window management strategies that apply directly to multimodal pipelines with minimal adaptation.

## When Unified Embeddings Are Not Enough

Unified embedding models are trained on broad data distributions, which means they can struggle with highly domain-specific visuals. Technical diagrams, medical imaging, and specialized charts often require domain-specific visual encoders or fine-tuned CLIP variants. In these cases, a hybrid approach works better: use a specialized visual encoder for the domain-specific image types, and fall back to unified embeddings for everything else.

Similarly, for tasks where [multimodal AI agents](/blog/multimodal-ai-agents-when-ai-sees-hears-and-acts) need to reason across many retrieved items rather than just retrieve and summarize, the retrieval quality ceiling becomes more apparent. Agentic workflows that iterate over retrieval results can partially compensate for lower recall, but it is not a substitute for a well-calibrated index.

## Key Takeaways

- Unified embedding models map multiple modalities into a single vector space, enabling cross-modal retrieval without separate indexes or score fusion heuristics.
- CLIP and its derivatives remain practical choices for image-text retrieval; newer models like Gemini Embedding 2 extend coverage to video and audio.
- Text queries surface both relevant text passages and relevant images from a single index when everything is embedded in a shared space.
- Per-modality score normalization is necessary before merging results across modalities, since raw cosine similarities are not calibrated between modality pairs.
- Dual-indexing images with both their visual embedding and a generated caption significantly improves recall for conceptual queries.
- Cap images passed to the generator at 3-5 to avoid context overflow and cost escalation; deduplicate images that appear via multiple retrieval paths.
- Plan for embedding model versioning from day one - re-indexing is expensive and images may not be available for re-processing on demand.
- Domain-specific visuals often require specialized encoders; unified embeddings work best as a generalist layer with domain models plugged in for specific content types.
